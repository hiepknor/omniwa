# OmniWA Workflow Catalog

## Purpose

This catalog defines the Phase 3.2 Application workflows for OmniWA.

Each workflow describes orchestration only. Business decisions remain in Domain aggregates, policies, specifications, domain services, and repository ports approved in Phase 2.

## Catalog Rules

- Workflow steps may load state, invoke Domain behavior, persist outcomes, create visible async work, and decide publication timing.
- Workflow steps must not define REST endpoints, DTOs, database schemas, queue implementation, provider implementation, or source code.
- Workflow decisions must reference approved Domain rules instead of re-stating or changing them.
- A workflow that accepts async work must make that work visible before returning an accepted outcome.
- Provider-native input must be translated before workflow orchestration.
- Secret and raw Confidential data must not be logged, sent to webhook consumers, or exposed in workflow outcomes.

## Catalog Constraints

- A workflow may call another workflow only through Application orchestration rules in `WORKFLOW_DEPENDENCIES.md`.
- No workflow owns Domain state directly.
- No workflow bypasses repository ports for persistence.
- No workflow exposes provider-native payloads to Domain, Webhook, Audit, or Observability.
- No workflow creates Domain Events outside aggregate roots.
- No workflow reports async acceptance unless the owner lifecycle or WorkerJob lifecycle is visible.
