# OmniWA Use Case Catalog

## Purpose

This document inventories OmniWA Application Layer use cases for Phase 3.1.

It does not define REST APIs, OpenAPI, DTOs, database schema, repository implementation, service implementation, provider implementation, queue implementation, or source code.

## Catalog Rules

- Use cases are product workflows, not endpoints.
- Inputs and outputs are application-level product concepts, not transport DTOs.
- Business rules are referenced from the frozen Domain Model.
- Domain Events listed are produced by aggregate roots; Application controls publication timing.
- Internal provider, worker, scheduler, and event-handler workflows still enter through Application use cases.
- Query use cases do not mutate Domain state and do not publish Domain Events.

## Actors

| Actor | Meaning |
| --- | --- |
| API Client | External product client invoking future interface boundaries. |
| Developer / Operator | Technical user operating OmniWA. |
| Admin | Actor allowed to perform privileged lifecycle, configuration, or recovery actions. |
| Provider Signal Handler | Application boundary receiving translated provider observations from provider adapters. |
| Worker Runtime | Runtime role requesting application work execution through approved boundaries. |
| Scheduler | Runtime role requesting periodic recovery, cleanup, or health workflows. |
| Application Event Handler | Internal handler reacting to Application-controlled publication decisions. |
| Monitoring System | External or internal monitoring consumer of safe health/telemetry queries. |
